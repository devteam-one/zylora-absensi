# Zylora — Project Audit (2026-06-28)

Read-only audit of the full repository (backend `server/api/**`, frontend
`src/app/**`, build/deploy/CI, packaging). No files were modified. Findings are
ordered by severity; two headline issues were verified empirically (geofence
bypass, payroll infinite loop).

Severity legend: **CRITICAL** exploitable now with serious impact · **HIGH**
serious but conditional · **MEDIUM** real risk / correctness gap · **LOW**
hardening / hygiene.

**Scope / method:** static review of all source + targeted empirical verification
(C1 confirmed with a live seeded exploit; H2 and H5 confirmed empirically/by grep).
`pnpm audit --prod` reported **no known CVEs** in the production dependency tree at
audit time. Not covered: a full running-system pentest, load testing, and the
SQLite data at rest.

---

## Remediation status (2026-06-28)

All six top findings addressed and verified (live exploit re-tests + smoke test 6/6, no
regression; frontend bundle builds clean — 2168 modules).

- **C1 fixed** — `checkGeo` coerces lat/lng and rejects non-finite/out-of-range
  (`attendance-core.mjs`). Verified: spoof coords → `400 NO_GPS`; valid → `201 hadir`.
- **H1 fixed (both halves)** —
  (a) `recordCheckin`/`recordCheckout` reject `loc.company_id !== emp.company_id` →
  `403 WRONG_COMPANY`.
  (b) `/api/public/location` now requires `?location=<id>` or `?company=<id>` (multi-tenant
  decision) — no more global cross-tenant row; missing scope → `400 SCOPE_REQUIRED`,
  unknown → `404`. `/api/me` now returns `companyId`; frontend display uses
  `VITE_LOCATION_ID`/`VITE_COMPANY_ID`, employee app & control panel pass their own
  `companyId`. Verified backend 5/5; display bundle builds.
- **H2 fixed** — UTC-safe date math in `payroll-core.mjs`. Verified: an 11-day leave
  expands in ~28 ms under `TZ=Asia/Jakarta` (previously an infinite loop).
- **H3 fixed** — `requireAuth` binds the JWT `sub`/`cid`/`role` to the stored session
  row and rejects unknown/missing roles (`middleware.mjs`). Verified: forged
  role-escalation token → `401 SESSION_MISMATCH`; normal login unaffected.
- **H4 fixed (per chosen scope)** — `electron`/`electron-builder`/`@capacitor/*` pinned
  exact in the binary `package.json`s; binary CI installs switched to `npm ci`;
  `permissions: contents: read` added to all 3 workflows. *Follow-ups documented in the
  workflows:* operator must generate & commit `package-lock.json` in `android-app/` and
  `desktop-app/` (one-time, needs network); root build can adopt
  `pnpm install --frozen-lockfile` for full reproducibility (needs CI testing).
- **H5 fixed** — removed the dead `db-pg.mjs` and `deploy/neon-schema.sql`; `server.mjs`
  refuses to start when `ZYLORA_DATABASE_URL`/`DATABASE_URL` is set (loud failure instead
  of silent SQLite-write trap); PG claims trimmed from `docs/LANJUTAN.md`. The unused
  `@neondatabase/serverless` dep can be dropped at the next lockfile regen.

Top-3 MEDIUM also fixed and verified:
- **M-print fixed** — `printSlip` now HTML-escapes every server/CSV value before
  `document.write` (`App.tsx`), closing the same-origin admin-token theft vector. Verified:
  `<script>`/`<img onerror>` payloads escaped to inert text; build compiles.
- **M-base fixed** — `apiUrlGuard` is now **fail-closed** (`vite.config.ts`): a production
  build without `VITE_API_URL` errors instead of silently baking the prod fallback; escape
  hatch `VITE_ALLOW_NO_API_URL=1` for intentional local builds. Verified: no URL → build
  fails; escape hatch / real URL → succeeds.
- **M4 fixed** — `http.mjs` resolves the client IP from the last `X-Forwarded-For` hop when
  `ZYLORA_TRUST_PROXY` is set (single trusted proxy), so per-IP rate-limit and `audit_logs.ip`
  work behind nginx; default still ignores the spoofable header. Verified via `audit_logs`:
  off → socket IP; on → `203.0.113.7` from XFF.

Still open (lower priority): scrypt-on-miss login timing, dynamic-QR concurrent-scan UX,
DB indexes/pagination, deactivated-employee `/api/me/*` reads, CORS default, and the LOW
list (money-as-float, negative-input validation, OTA `javascript:` link, etc.).

## CRITICAL

### C1 — Geofence is bypassable with non-numeric GPS (NaN comparison) → fake "present" from anywhere
`server/api/lib/attendance-core.mjs:106-113` (`checkGeo` + `distanceM`), reached from `routes/employee.routes.mjs` check-in.

`lat`/`lng` come straight from the request body. `checkGeo` asserts they are
non-null, then does `if (distanceM(...) > loc.radius_m) throw`. A **non-null but
non-numeric** value (`"lat":"x"`) passes the null check, makes `distanceM` return
`NaN`, and `NaN > radius` is **`false`** → the radius check is skipped and the
row is recorded as `hadir`. **Verified:**

```
lat/lng="spoof" -> dist=NaN  (dist>radius? false)  => BYPASSED
```

Full exploit: `GET /api/public/location` (no auth) returns a currently-valid
location token → `POST /api/me/checkin {location_token, lat:"x", lng:"x"}` passes
identity (token), QR validity (fresh token), and geofence (NaN). The geofence is
the *only* control preventing remote fake attendance — and it is fully defeated.

**Verified live (seeded API, throwaway DB):** login EMP001 → `GET /api/public/location`
(unauthenticated) returned exact coords `-6.2088, 106.8456` + a valid token →
`POST /api/me/checkin {lat:"spoof", lng:"spoof"}` returned **`201`** with
`{"status":"hadir","location":"Kantor Pusat Jakarta"}`. Recorded present with garbage
coordinates. (The same call also confirms H1's unauthenticated GPS+token leak.) The
flow reaches `checkGeo` uncoerced — `validate.mjs` has no numeric coercion and
`requireFields` only checks `location_token`.

**Fix:** coerce and strictly validate before distance — reject unless
`Number.isFinite(lat) && Number.isFinite(lng) && lat∈[-90,90] && lng∈[-180,180]`.
Necessary but not sufficient (see M1): GPS is self-reported and the exact
coordinates are leaked by `/api/public/location`, so for real assurance bind
check-in to a registered device / signed attestation, or document the geofence
as advisory.

## HIGH

### H1 — `/api/public/location` is globally unscoped → cross-tenant data leak + cross-tenant check-in + multi-location broken
`server/api/routes/public.routes.mjs:14-18`; compounded by `lib/attendance-core.mjs` `recordCheckin`/`resolveLocation`.

```sql
SELECT lc.*, l.name, l.lat, l.lng, l.radius_m
FROM location_codes lc JOIN locations l ON l.id = lc.location_id
WHERE lc.status = 'active'
ORDER BY (lc.type='qr_dynamic') DESC, lc.created_at DESC LIMIT 1
```

No company/location filter, no auth — returns the single globally-newest active
code in the whole DB. Consequences: (a) any tenant with ≥2 locations can only
ever serve one location's display screen; (b) in the documented multi-tenant
mode, one tenant's location name, **exact GPS**, radius and a live scannable
token are served to every anonymous caller; (c) `recordCheckin` never asserts
`loc.company_id === emp.company_id`, so an employee of company A can check in
against company B's location.

**Fix:** scope by an explicit `locationId`/company context instead of global
`LIMIT 1`; assert `loc.company_id === emp.company_id` in the check-in path.

### H2 — `nextDay()` infinite-loop DoS under any positive-UTC-offset host timezone (incl. Asia/Jakarta)
`server/api/lib/payroll-core.mjs:15`, used by leave-date expansion at line 51.

```js
const nextDay = (ds) => { const dt = new Date(ds + "T00:00:00"); dt.setDate(dt.getDate()+1); return dt.toISOString().slice(0,10); };
```

`new Date("…T00:00:00")` parses as **local** time; `.toISOString()` converts back
to UTC, undoing the +1 on positive-offset hosts. **Verified:**

```
TZ=Asia/Jakarta  nextDay(2026-06-15) = 2026-06-15   <<< never advances → infinite loop
TZ=UTC           nextDay(2026-06-15) = 2026-06-16   ok
```

The loop `while (d <= end) { …; if (d===end) break; d = nextDay(d); }` never
terminates for any approved leave spanning >1 day. The product is Indonesian and
the default/seed TZ is `Asia/Jakarta` (UTC+7) — so a host set to its own
timezone makes any `POST /api/payroll/run` (and `tools/rerun-payroll.mjs`) pin a
CPU core forever on the single-threaded server → full DoS. The seed data itself
(EMP006 multi-day leave) triggers it.

**Fix:** UTC-safe date math — parse with `"…T00:00:00Z"` and use
`getUTCDate()`/`setUTCDate()`, or add `86400000` ms to a `Z`-anchored date. Same
fragility in `daysInMonth`.

### H3 — JWT session is validated for liveness only; token claims (`role`, `cid`, `sub`) are never bound to the stored session row
`server/api/lib/middleware.mjs:20-33`.

`requireAuth` reads back only `revoked` from `sessions` and then trusts
`role`/`cid`/`sub` straight from the JWT, even though login persists
`subject_id`/`company_id`/`subject_type`. The session row gates *jti liveness*,
not *identity*. So anyone able to sign a JWT (i.e. who knows `ZYLORA_SECRET`)
can log in as their own employee to get a live `jti`, then mint a second token
reusing that `jti` with `role:"control"` and any `cid` → full cross-tenant admin.
Defense-in-depth (needs the secret to exploit), but it converts any secret leak
into total takeover.

**Fix:** `SELECT subject_type, subject_id, company_id, revoked` and reject unless
they match the token claims; build `ctx.auth` from the DB row, not the token.
Also: `const role = payload.role || "control"` (line 23) fails open to the
privileged role — treat missing/unknown role as auth failure.

### H4 — CI builds signed, user-facing binaries from unpinned, lockfile-less installs
`.github/workflows/android-apk.yml`, `desktop-control.yml`; `desktop-app/package.json`, `android-app/package.json`.

The release pipelines emit **signed APK/AAB + desktop installers shipped to end
users**, but install deps with `npm install` (never `npm ci`); the root has only
`pnpm-lock.yaml` (npm ignores it), `desktop-app/` has **no lockfile at all**, and
`electron`/`electron-builder`/`@capacitor/*` float on caret ranges. Builds are
non-reproducible and a single compromised in-range transitive version lands
inside a signed binary on users' devices.

**Fix:** commit a real lockfile per build root, switch CI to
`npm ci`/`--frozen-lockfile`, pin `electron`/`electron-builder`/`@capacitor/*`
exact.

### H5 — Postgres backend (`db-pg.mjs`) is non-functional: never wired, sync/async mismatch, reserved-word SQL
`server/api/lib/db-pg.mjs` (whole file).

**Verified:** `grep -rn db-pg server/` → zero references; no `lib/db/` selector
directory; `ZYLORA_DATABASE_URL` is referenced *only inside `db-pg.mjs` itself*.
Every module imports SQLite `./db.mjs` directly, so `ZYLORA_DATABASE_URL` has **no
effect**. Even if wired: its `get/all/run` are async (return Promises) while all
call sites use results synchronously → every route breaks; and reserved columns
`start`/`end`/`interval` are emitted unquoted (SQLite tolerates, Postgres rejects).

The HIGH-severity risk is not "feature missing" — it's a **silent data-durability
trap**: an operator who sets `ZYLORA_DATABASE_URL` expecting managed Neon Postgres
gets an app that silently keeps writing to the gitignored local SQLite file. Their
real data isn't where they think it is and isn't covered by managed-DB backups.

**Fix:** either remove `db-pg.mjs` + `deploy/neon-schema.sql` + the PG docs, or
actually implement it (selector, `await` everywhere, quote/rename reserved
columns).

## MEDIUM (grouped)

**Attendance integrity**
- **M1 Remote attendance fraud is structurally possible** even after C1's fix: GPS is
  self-reported with no attestation, `/api/public/location` leaks exact coords, and
  the `devices` (registered-device) table is never consulted in check-in. Locations
  with null coords skip geofencing entirely. Treat geofence as advisory or add
  device binding / signed location nonce. (`attendance-core.mjs`, `employee.routes.mjs`)
- **M3 Dynamic-QR single-use serial breaks concurrent kiosk scans:** every scan bumps
  the serial and validity requires exact match, so the 2nd+ employee scanning the
  same displayed QR during a morning rush gets `400 "QR already scanned"` until the
  display re-polls. Decouple anti-replay from per-scan bump for shared displays.
  (`attendance-core.mjs:128-132,165`)
- Check-out skips geofence and stores no location/coords (`employee.routes.mjs`).

**Auth / rate-limiting**
- **M-enum Login user-enumeration via scrypt timing oracle:** non-existent accounts
  return immediately, existing ones incur full scrypt. Run a dummy scrypt on the
  miss path for constant time. (`auth.routes.mjs:31-35`, `employee.routes.mjs:35-38`)
- **M4 Rate-limit + audit IP useless behind the documented nginx proxy:** `ctx.ip` is
  the raw socket peer = `127.0.0.1` for every request, so the per-IP login bucket
  becomes one global counter (one attacker / morning load locks out everyone) and
  every `audit_logs.ip` is `127.0.0.1`. Per-account bucket also keys on the *submitted*
  identifier (employeeId vs email doubles the allowance) and resets on every restart
  (in-memory). Trust a proxy header behind a `ZYLORA_TRUST_PROXY` flag; key per-account
  on the resolved id. (`http.mjs:132`, `middleware.mjs:67-84`)
- **Dev fallback secret is warn-only outside prod:** hard `exit(1)` only fires when
  `NODE_ENV/ZYLORA_ENV === production`; any launcher that omits that boots on the
  public repo constant `zylora-dev-secret-change-me` (signs JWTs *and* QR HMACs).
  Refuse to start on a weak/fallback secret regardless of the prod flag.
  (`security.mjs:21-46`) — amplifies H3.
- **CORS defaults to `*`** unless `ZYLORA_CORS_ORIGIN` set (low-impact for a bearer API,
  but should default-deny in prod). (`http.mjs:17`)
- Deactivated employee keeps reading `/api/me/payslips|attendance|leave` (those skip
  the `meEmployee()` active-check); password/PIN change & deactivation don't revoke
  existing sessions (token valid up to 8–12h). (`employee.routes.mjs`)

**Payroll correctness**
- **M-fx Multi-currency / exchange rates are never applied** — `exchange_rates` is a
  dead table, `payslips.currency` is just the company base-currency label slapped over
  whatever number was entered. A salary entered in a different currency is silently
  mislabeled. Either remove the FX UI/table or add per-employee `salary_currency` +
  actual conversion. (`payroll.routes.mjs`, `payroll-core.mjs`)
- Payroll "today" cutoff uses the global `DEFAULT_TZ`, not the company TZ → off-by-one
  absent-day deductions for non-default-TZ tenants. (`payroll-core.mjs:57`)
- No base-salary proration for mid-period joiners; re-running a past period drops
  now-deactivated employees (their historical payslip vanishes and can't be
  regenerated). (`payroll-core.mjs`, `payroll.routes.mjs`)

**Frontend**
- **M-print Payslip print builds HTML from unescaped server/CSV data into a same-origin
  `window.open("")` via `document.write`** → injected `<img onerror>`/`<script>` runs and
  can read `localStorage["zylora.control.token"]` (live 8h admin JWT). Escape all
  interpolated values or render into a sandboxed iframe. (`App.tsx:2059-2073`)
- **M-base `api.ts` silently defaults to the production backend** when `VITE_API_URL` is
  unset, and `apiUrlGuard` only warns unless `VITE_REQUIRE_API_URL=1` (which
  `build-frontends.sh` does NOT set). A forgotten env var ships a bundle that mutates
  prod. Make the build fail by default; default dev to loopback. (`api.ts:9-10`,
  `vite.config.ts:51-72`)
- JWTs in `localStorage` (XSS-exfiltratable, amplifies M-print); `remember` defaults
  to `true`. Prefer HttpOnly cookie or in-memory; default `remember` to `false`.
- 401/expired token is shown as **"Server unreachable"** and never forces re-login
  (employee poll swallows the 401 and keeps a stale logged-in UI). Distinguish
  401/403 and drop to login. (`App.tsx:2389-2392`, `470`)

**Ops / deploy / CI**
- Destructive ops tools (`tools/cleanup-demo.mjs`, `cleanup-zylora1.mjs`,
  `rerun-payroll.mjs`) run irreversible prod `DELETE`s with no `--confirm`/`--dry-run`/
  prod-guard; `rerun-payroll` with no `--company` wipes payroll for **all** tenants.
  Add confirmation + dry-run + auto-backup.
- `ZYLORA_SECRET` transits as an inline process argument over SSH during deploy
  (visible in the remote process table). Pass via stdin / 600-mode env file.
  (`deploy/deploy.sh:57-59`)
- No `permissions:` block on the secret-bearing CI workflows (broad default token).
- `curl … nodesource | sudo bash` with no integrity check (root RCE if MITM'd).
- `apiUrlGuard` is presence-only (a loopback/typo URL passes). Validate the value.
- Missing secondary indexes (`attendance(company_id,date)`, `payslips(run_id)`, …) and
  several unbounded admin list queries + N+1 recap/payroll-runs → scans grow with data.
  (`db.mjs`, `attendance.routes.mjs`, `employees.routes.mjs`, `payroll.routes.mjs`)

## LOW (selected)

- Money stored as floating-point (`REAL`/`DOUBLE PRECISION`); `Math.round` per line
  keeps IDR exact today but prefer integer minor units.
- Negative/zero numeric inputs accepted silently (negative deduction increases pay; net
  can go negative). Validate `>= 0`.
- `POST /api/locations` doesn't validate `radius_m`/`lat`/`lng` at create (the PUT does).
- Empty `catch{}` blocks swallow errors (`employees.routes.mjs`, `db.mjs` migrations).
- Schema drift: `deploy/neon-schema.sql` is a hand copy of `db.mjs`; runtime ALTERs not
  reflected; neither has the M-index set.
- Leftover Indonesian strings in the English-migrated UI + `index.html lang="id"`.
- OTA-manifest `<a href={upd.url}>` not scheme-validated (`javascript:` link) and lacks
  `rel="noopener"`. (`App.tsx:341`)
- Dead `useDynamicQR` in control panel burns an always-on 1s interval rendering nothing.
- Unhandled promise rejections on delete actions (no `.catch`, no error toast).
- `package.json` `pnpm.overrides` (the `rollup@4.61.3` pin) is **silently ignored** by
  the installed pnpm version ("The 'pnpm' field … is no longer read"), so the intended
  transitive pin is not actually enforced. Move it to the supported `overrides` location.
- CI actions pinned to mutable tags (`@v7`) not SHAs; prod IP/instance-id/hostname
  committed in deploy scripts & `api.ts`; admin password passed via argv in
  `register-admin.mjs`; plaintext un-encrypted DB backups co-located with the DB;
  `desktop-app/README.md` stale ("auto-login demo" — code is explicit-login now).

## Done well (verified, not assumed)

- **No SQL injection** anywhere — all queries parameterized; dynamic identifiers come
  from hardcoded allowlists. Same in the (unused) PG variant.
- **JWT has no alg-confusion** — signature recomputed as HMAC-SHA256 ignoring the header
  `alg`; expiry enforced; **constant-time** compares with length guards in both password
  and JWT verify.
- **Password/PIN hashing**: salted `scryptSync(...,64)` with per-credential 16-byte salt.
- **RBAC consistently applied** on the authenticated surface; control routes gated by
  `requireControl`, `/api/me/*` by `requireEmployee`, cross-role → 403; consistent
  multi-tenant `company_id` scoping on admin CRUD (the gaps are the *public* check-in
  paths, H1, not the admin routes).
- **No password-hash leakage** (serializers expose `has_pin` only); generic 500 on
  unhandled errors (no stack-trace leak); 5 MB body cap.
- `ZYLORA_SECRET` missing/weak/<16-char is a **fatal exit in production** (and enforced
  via `${ZYLORA_SECRET:?}` in deploy); seed only on `ZYLORA_SEED=1`; **no source maps
  shipped**; **no postinstall scripts**; `.env.production.local` not tracked and holds
  no secret.
- Deploy scripts use `set -euo pipefail`, quote vars, `trap cleanup EXIT`; systemd
  hardening (`NoNewPrivileges`, `ProtectSystem=full`, `PrivateTmp`); binds `127.0.0.1`
  behind nginx+TLS.
- **Electron hardening correct**: `contextIsolation:true`, `nodeIntegration:false`,
  `loadFile`, external links denied to OS browser; Capacitor `allowMixedContent:false`,
  least-privilege permissions (camera+location only).
- Haversine correct & unit-consistent; overnight-shift lateness handled; attendance
  recording is timezone-aware via `Intl` with the company zone; payroll re-run is atomic
  (`tx()`); `UNIQUE(employee_id,date)` prevents duplicate attendance; approved-leave
  blocks contradictory check-ins.
- Frontend: no CSRF surface (bearer only), no client-side security decisions (desktop-
  only control gate is UX, not a boundary), logout revokes server-side + validates saved
  token on restore, polling intervals cleaned up and paused during input.

## Suggested remediation order

1. **C1** geofence NaN validation (one-function fix, kills the worst exploit).
2. **H2** `nextDay` UTC-safe date math (prevents prod payroll DoS in the product's own TZ).
3. **H1** scope `/api/public/location` + assert company match on check-in.
4. **H3** bind session row to token claims + reject missing role; refuse weak secret
   regardless of prod flag.
5. **H4** lockfiles + `npm ci` + exact pins in the signed-binary CI pipelines.
6. **H5** decide Postgres: implement properly or remove the dead path + docs.
7. Medium cluster: payslip-print escaping, build fail-on-missing-API-URL, proxy-aware
   rate-limit/audit-IP, scrypt-on-miss timing, FX (apply or remove), ops-tool guards.
